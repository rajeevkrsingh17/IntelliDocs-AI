# ADR-003: Chunk-Level Source Citation for Answer Transparency

## Status

**Accepted** — Implemented in production

## Date

2026-07-10

## Context

A fundamental risk in any Retrieval-Augmented Generation (RAG) system is **hallucination** — the LLM generating plausible-sounding but factually incorrect content not present in the source documents. In business and research contexts (legal contracts, financial reports, technical specifications), uncited AI-generated answers carry a risk of being trusted without verification.

Additionally, even when the LLM answer is correct, a user has no way of knowing *which part of which document* the answer came from. This creates a transparency and trust problem: users cannot verify answers, cannot navigate to the source passage, and cannot judge whether the retrieved context was actually relevant to their question.

The solution must:
1. Attach source information (document name, page number) to every AI-generated answer at display time.
2. Not require changes to the LLM prompt itself — citation data should come from retrieval metadata, not be "invented" by the model.
3. Be displayable in both the Streamlit and React frontend without duplication of logic.

## Decision

Every document chunk stored in ChromaDB includes the following metadata fields at ingestion time:

```python
metadata = {
    "document_name": str,   # Original filename (e.g., "annual_report_2025.pdf")
    "document_type": str,   # File extension (e.g., "pdf")
    "chunk": int,           # Sequential chunk index within the document
    "page": int,            # Estimated page number (calculated from chunk position)
    "upload_time": str,     # ISO timestamp of when the document was uploaded
}
```

At query time, the top-k retrieved chunks are returned from `scripts/search.py` **with their full metadata dictionaries**. The `/query` FastAPI endpoint then packages this alongside the LLM-generated answer in the response payload:

```json
{
  "answer": "The net profit margin for FY2025 was 18.4%.",
  "sources": [
    {
      "document": "annual_report_2025.pdf",
      "page": 12,
      "chunk": 47,
      "score": 0.91
    }
  ],
  "model_used": "gemini-2.0-flash"
}
```

Both the React frontend and Streamlit UI render these citations as visible source cards below the generated answer — explicitly labelling each citation with the document name and page number.

## Alternatives Considered

### LLM Self-Citation (Prompt the model to include citations)
- **Approach:** Instruct the LLM in the system prompt to include `[Source: document_name, page X]` markers inside the generated answer text.
- **Why rejected:** LLMs frequently hallucinate citation numbers — they may cite page 4 when the actual passage was on page 12. Citations generated by the model are not grounded in any real metadata; they reflect the model's own internal reasoning, not the retrieval system's verified metadata. This would actively mislead users.

### No Citations (Answer Only)
- **Approach:** Return only the synthesized answer text, omitting source information entirely.
- **Why rejected:** Without citations, the system provides no mechanism for users to verify answers. For the target use case (professionals using dense document corpora), this is unacceptable. It also removes the core value proposition of RAG over a simple chatbot — the ability to point users directly to the source passage.

### End-of-Answer Footnote (Appended Text)
- **Approach:** Append a plain-text citation block to the end of the answer string (e.g., "Sources: report.pdf, p.12").
- **Why not chosen:** This conflates the citation display with the answer text in the database. The structured JSON approach (separate `sources` field) is cleaner: the frontend can choose how to render citations independently from the answer text, and the data format is machine-readable for future features (e.g., "show me all answers that cited this page").

## Consequences

### Positive
- **Verifiable answers:** Users can immediately navigate to the cited page in their document to verify the AI answer against the source passage.
- **Trust signal:** Visible citations signal to users that the system is retrieval-grounded, not hallucinating from the model's training data.
- **Future-proof schema:** The metadata schema (document, page, chunk, score) is extensible — adding a `confidence_score`, `highlight_offset`, or `section_title` field in future requires no schema migration in ChromaDB.
- **No prompt engineering required:** Citations derive from ChromaDB metadata, not from instructing the LLM, which makes the citation data reliable regardless of the LLM tier used (including the offline mock fallback).

### Negative
- **Page approximation, not exact location:** The `page` field is *calculated* from the chunk's sequential position within the document (using an estimated words-per-page ratio), not extracted from a precise PDF coordinate. For documents with very uneven page densities (e.g., pages dominated by large tables or images), the page number may be off by ±1. This is disclosed as a known limitation.
- **Increased response payload size:** Including source metadata adds approximately 200-400 bytes per cited chunk to each API response. At the scale of this project (5-10 retrieved chunks per query), this is negligible.
